# GPU Support Progress

Last updated: 2026-05-05

## Objective

Implement an opt-in GPU path for protein `hmmsearch --gpu`, without CPU fallback for requested GPU stages, then evaluate later pipeline stages one by one and keep only stages that are suitable for GPU and fast enough on representative benchmarks.

## Current State

- TODO exists in `agents_docs/gpu-support-todo.md`.
- An opt-in CUDA MSV implementation is present for protein `hmmsearch --gpu` on protein `dsqdata` target input built with `hmmseqdb`.
- The GPU path also computes the biased-composition filter score in CUDA batches, reusing the sequence batch already uploaded for MSV.
- `origin/cuda` remains historical reference material only.
- Build detection is wired through `configure --enable-cuda` and `--with-cuda-arch=sm_XX`.
- `nvcc` and CUDA runtime libraries are present in the environment.
- `hmmseqdb` builds existing Easel protein `dsqdata`; `hmmsearch --gpu` rejects ordinary FASTA target input.
- The Easel `dsqdata` chunk-sizing change is applied at build time from `patches/easel-dsqdata-open-sized.patch`, not edited in place in the submodule.

## Required Deliverables

1. CUDA runtime and MSV/bias module under `src/`.
2. `hmmsearch --gpu` opt-in CLI path.
3. No fallback to CPU when GPU mode is explicitly requested.
4. Benchmark and timing check proving GPU speedup versus CPU MSV on the same fixture set.
5. Progress updates recorded here as work lands.

## Verification Targets

- CUDA MSV parity against CPU `p7_MSVFilter()`.
- CUDA bias filter parity against CPU `p7_bg_FilterScore()` at the search-hit level and in pipeline counters.
- Overflow/high-score parity.
- End-to-end `hmmsearch --gpu` versus CPU `hmmsearch`.
- Timing comparison that records wall-clock and GPU kernel time.
- Error handling for unsupported builds/devices/databases/options.

## Timing Requirement

GPU work must not be treated as complete until timing evidence shows a real speedup on at least one representative benchmark fixture.

Minimum timing evidence to record:

- CPU MSV wall time.
- GPU MSV wall time.
- GPU kernel elapsed time.
- Host/device transfer time.
- CUDA bias H2D/kernel/D2H time when bias is enabled.
- Batch size used.
- Number of CUDA search batches launched when evaluating load/search batch packing.

## Remaining Scope

- TODO validation categories for a true `dsqdata` v2 length-index extension remain open; current implementation uses existing Easel dsqdata and per-sequence lengths after chunk unpacking.
- Current `hmmsearch --gpu` uses dsqdata input with GPU replacing MSV and computing CUDA bias scores by default, but the final bias/F1 boundary is CPU-compatible for exact hit parity: CPU `p7_bg_FilterScore()` supplies the final bias score for GPU MSV survivors, and CPU `p7_MSVFilter()` can rescue bias-boundary rejects through the optimized CPU SSV shortcut. Experimental opt-in CUDA Viterbi, Forward score-prefilter, and Forward/Backward parser-state paths now compose on the prepared all-13 profmark set and a separate 12-query broader profmark sample with exact final hit parity, clean Viterbi/Forward score diagnostics, and a same-build speedup versus CPU in non-compare mode. They remain opt-in because exhaustive validation beyond these samples and raw parser scale-row diagnostics are still open.
- Current `hmmsearch --gpu` has separate load/search batch controls and can pack several dsqdata load chunks into a single CUDA filter batch. Profiling shows larger search batches can reduce kernel launches and kernel time, but the current default remains 32,768 sequences / 8M residues because the all-13 profmark wall time did not improve at 12M or 16M search batches.
- GPU is not universally faster, but the current tuned dsqdata path shows end-to-end speedups on representative profmark samples with exact all-13 prepared-profmark hit parity. The biggest remaining efficiency bottleneck is CPU survivor continuation after GPU MSV/bias and the CPU-compatible bias/MSV boundary checks; dsqdata read/unpack and H2D/D2H transfer are comparatively small in the current measurements.
- Moving the null filter to GPU is unlikely to matter because null time is tiny. Bias remains useful as a CUDA batch computation only because it reuses the MSV batch upload, but exact parity currently requires CPU-compatible boundary checks. Further bias optimization should preserve the current zero-delta all-13 behavior before trying to reclaim the CPU boundary-check cost.
- Viterbi remains a logical bottleneck. The current score-compare harness is clean on the all-13 prepared profmark query set and the broader 12-query sample after per-length special transition handling, and the current non-compare runs with Viterbi/Forward/Backward enabled improve aggregate wall time versus CPU while preserving exact final hit parity.
- External dense-HMM CUDA implementations are not a shortcut for this work. HMMER Viterbi/Forward/Backward implementation needs a Plan7-specific parity harness and state-transfer design; see `agents_docs/cuda-hmm-reference.md`.
- Forward/Backward is partially suitable, but not as a simple isolated score offload unless the implementation explicitly handles the CPU matrix-state dependency. The post-Forward helper split makes a narrow score prefilter possible, but the five-query and all-13 upper bounds are modest and any prototype must be compared against the current MSV+bias baseline, including the cost of rerunning CPU Forward for F3 survivors.
- The current `--gpu-fwd-prefilter` experiment has a prefix-scan path only for very short profiles (`M <= 256`), a CPU Forward gray zone near F3, and CPU continuation for longer profiles. The prefix-scan race found in live audit has been fixed and the all-13 compare run has zero `CUDAFWD` score/pass diagnostics, but Forward remains a score prefilter only unless `--gpu-fb-parser` is also enabled to provide parser state.
- The current `--gpu-fb-parser` prototype is no longer just scalar parser scaffolding: it has batched F3-survivor handoff, block-parallel short-profile parser Forward and Backward kernels, all-13 prepared-profmark timing evidence, broader 12-query timing evidence, and bounded posterior/domain input differences in both compare runs. It remains experimental rather than default because raw `p7X_SCALE` row differences still appear in compare diagnostics and exhaustive validation is still pending.
- CUDA Viterbi score comparison is parity-clean on the all-13 prepared profmark query set and the broader 12-query sample after per-length special transition handling. It is still opt-in/default-off while exhaustive validation and default policy are pending.
- Backward, domain definition, null2, hit storage, thresholding, and output should stay on CPU for now. Current measurements show Backward and null2/output are tiny, and domain work happens after Forward has already reduced the survivor set to a small number of sequences.
- Sensitivity parity on the prepared all-13 profmark set is now exact because the GPU path includes a CPU-compatible bias/MSV boundary check. A fully CUDA-native SSV equivalent remains deferred: CPU SSE can accept `p7_SSVFilter()` results before full MSV, while GPU v1 still computes full MSV and relies on the CPU boundary rescue for exact parity.
- A direct CUDA diagonal implementation over the CPU `sbv` profile was tested and reverted because it did not reproduce `p7_SSVFilter()` behavior. Future SSV work should map the CPU banded algorithm more faithfully if the CPU boundary rescue cost needs to be removed.
